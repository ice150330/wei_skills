# Complexity Assessor

> 🔍 **首要技能** - 每个用户请求都应该先经过复杂度评估

## 目的

避免"大炮打蚊子"问题：
- **简单任务** (`print("hello")`) → 直接回答，不加载工作流
- **中等任务** (添加一个API端点) → 轻量级流程，1-2个角色
- **复杂任务** (开发完整电商平台) → 完整工作流，7个角色

## 复杂度评估标准

### Level 1: SIMPLE (简单)

**判定条件** (满足任一):
- 单文件修改 (< 50 行)
- 单行命令执行
- 简单问答/解释
- 代码片段解释
- 快速修复 (typo, 小bug)
- 已有明确解决方案

**示例**:
```
"怎么print输出？"
"修复这个拼写错误"
"解释一下这段代码"
"运行一下这个测试"
"git status 看一下"
```

**处理方式**:
- ✅ 直接回答
- ❌ 不启动工作流
- ❌ 不创建计划
- ⚡ 响应时间: < 10秒

---

### Level 2: MODERATE (中等)

**判定条件** (满足任一):
- 2-5个文件的修改
- 单一功能模块开发
- 现有功能的扩展
- 组件/页面开发
- API端点开发
- 需要设计决策但没有架构变化

**示例**:
```
"给登录页面添加验证码功能"
"创建一个用户列表API"
"优化这个组件的性能"
"添加一个数据导出功能"
"设计一个新的表单组件"
```

**处理方式**:
- ✅ 启动 simple-workflow
- ✅ 涉及 1-2 个角色 (如 Frontend + QA)
- ✅ 轻量级计划 (3-5个任务)
- ⚡ 响应时间: 1-5分钟规划，然后执行

---

### Level 3: COMPLEX (复杂)

**判定条件** (满足任一):
- 全新项目开发
- 多模块系统设计
- 架构级别的变更
- 需要多个角色的协作
- 涉及设计+前端+后端+部署
- 不确定需求，需要探索

**示例**:
```
"开发一个电商平台"
"重构整个认证系统"
"设计一个实时聊天系统"
"从零开始创建一个博客"
"优化系统架构以支持百万用户"
```

**处理方式**:
- ✅ 启动完整 project-dev-workflow
- ✅ 涉及 5-7 个角色
- ✅ 详细计划 (Phase 1-4)
- ✅ 设计文档和架构规划
- ⚡ 响应时间: 10-30分钟规划

---

## 快速评估流程

```
用户输入
    ↓
[1秒快速扫描]
├── 单行代码/命令? → SIMPLE
├── 单文件修改? → SIMPLE
├── 需要解释/问答? → SIMPLE
    ↓
[3秒深入分析]
├── 修改范围 > 5个文件? → COMPLEX
├── 涉及多个模块? → COMPLEX
├── 全新功能开发? → MODERATE/COMPLEX
├── 架构设计? → COMPLEX
    ↓
[复杂度判定]
→ SIMPLE: 直接执行
→ MODERATE: simple-workflow (1-2角色)
→ COMPLEX: full-workflow (完整7角色)
```

## 评估检查清单

### 简单任务指标 ✅
- [ ] 修改范围 < 50 行代码
- [ ] 不涉及架构决策
- [ ] 已有明确实现方式
- [ ] 不需要协调多个组件
- [ ] 用户期望立即完成

### 中等任务指标 ⚖️
- [ ] 2-5个文件受影响
- [ ] 单一模块内的功能
- [ ] 需要设计但不复杂
- [ ] 可能需要1-2个角色
- [ ] 可以在1-2小时内完成

### 复杂任务指标 🏗️
- [ ] 全新项目或大型重构
- [ ] 多模块协调
- [ ] 需要架构设计
- [ ] 涉及多个技术栈
- [ ] 需要长期规划 (> 1天)

### SIMPLE 响应
```
这是一个简单任务，我直接帮你处理：

[直接给出答案/代码/命令]

✅ 完成
```

### MODERATE 响应

```
🎯 涉及角色: Frontend + QA
📋 计划:
1. 实现功能
2. 编写测试
3. 代码审查

[开始执行...]
```

```

Phase 2: 开发 (Frontend + Backend + QA并行)
Phase 3: 质量保证 (QA深度审查)

[开始详细规划...]
```

## 边界情况处理

### 用户说"简单"但实际复杂
```
用户: "简单，帮我做个电商平台"

评估: 尽管用户说"简单"，但电商平台涉及:
- 多个模块 (商品、订单、支付、用户)
- 数据库设计
- 前后端开发
- 部署运维

判定: COMPLEX → 启动完整工作流
解释: "虽然你说简单，但电商平台涉及多个模块，我建议使用完整工作流来确保质量"
```

### 用户说"复杂"但实际简单
```
用户: "这个很复杂，需要重构整个登录流程"

评估: 实际只是:
- 修改一个函数
- 更新验证逻辑
- 修改2-3个文件

判定: MODERATE → 轻量级流程
解释: "这个修改范围不大，我用标准流程快速完成"
```

## 动态调整

如果在执行过程中发现复杂度评估错误：

```
开始执行 MODERATE 任务...
    ↓
发现需要修改10+个文件
涉及数据库迁移
需要API版本兼容
    ↓
自动升级: MODERATE → COMPLEX
通知用户: "这个任务比预期复杂，我切换到完整工作流"
```

## 使用示例

### 示例1: 简单任务
```
用户: "Python怎么写Hello World"

评估:
- 单语句
- 已有明确答案
- 不需要规划

判定: SIMPLE

响应:
print("Hello World")
```

### 示例2: 中等任务
```
用户: "给现有API添加分页功能"

评估:
- 修改2-3个文件
- 单一功能点
- 需要测试

判定: MODERATE

响应:
🎯 启动标准工作流 (Backend + QA)
1. 修改API添加分页参数
2. 更新数据库查询
3. 编写测试用例
4. 代码审查
```

### 示例3: 复杂任务
```
用户: "开发一个类似Notion的笔记应用"

评估:
- 全新项目
- 多模块 (编辑器、协作、存储)
- 需要完整架构

判定: COMPLEX

响应:
🏗️ 启动完整7角色工作流
Phase 1-4 完整流程...
```

## 关键原则

1. **宁可低估，不要高估**: 简单任务评估错误影响小，复杂任务评估错误会导致返工
2. **5秒规则**: 5秒内必须做出复杂度判定
3. **用户沟通**: 明确告知用户为什么选择这个复杂度级别
4. **动态调整**: 执行中发现复杂度变化，及时调整
5. **尊重用户**: 如果用户明确要求简单处理，可以降级（但要提醒风险）
