# 根本指令

**此技能优先级最高，必须在每次对话开始时加载并执行。**

所有其他技能执行前，必须先遵守以下根本指令。如果其他技能与根本指令冲突，以根本指令为准。

---

## 一、语言规则（中文优先）

### 强制规则

**所有输出必须使用中文**，包括但不限于：

| 场景 | 语言要求 |
|------|----------|
| 对话回复 | 中文 |
| 代码注释 | 中文（不影响代码功能的部分） |
| 计划书/报告 | 中文 |
| 终端说明文字 | 中文 |
| 提交信息(commit message) | 遵循项目惯例，主体用中文，type用英文 |
| 错误解释 | 中文 |
| Todo列表内容 | 中文 |
| 选项说明 | 中文 |
| 上下文压缩 | 中文 |

### 例外

以下场景不受中文限制：
- 代码标识符（变量名、函数名、类名）用英文
- Git commit type（feat/fix/docs等）用英文
- 配置文件键名用英文
- 第三方库的API调用用英文
- 已有英文项目中的文件名用英文
- 正则表达式、SQL语句等技术内容保持原有语言

### 代码注释规范

```python
# 正确：中文注释解释意图
def calculate_total(items):
    """计算所有商品的总价，包含税率和折扣"""
    subtotal = sum(item.price for item in items)
    tax = subtotal * TAX_RATE
    return subtotal + tax

# 错误：英文注释
def calculate_total(items):
    """Calculate total price of all items including tax and discount"""
    subtotal = sum(item.price for item in items)
    tax = subtotal * TAX_RATE
    return subtotal + tax
```

## 二、项目结构规范

### 强制规则

**每个项目都必须有清晰的结构，文件归入对应目录，不得随意放到根目录。**

### 标准项目结构模板

```
项目根目录/
├── docs/                    # 文档
│   ├── plans/               # 设计计划书
│   │   └── 2026-04-12-功能名-design.md
│   ├── reports/             # 分析报告
│   │   └── 2026-04-12-代码审查报告.md
│   └── api/                 # API 文档（如适用）
├── src/                     # 源代码
├── tests/                   # 测试文件
├── config/                  # 配置文件
├── scripts/                 # 工具脚本
├── plan/                    # 简短计划（替代方案，与docs/plans二选一）
└── README.md                # 项目说明
```

### 文件归位规则

| 文件类型 | 必须放到 | 禁止放到 |
|----------|----------|----------|
| 计划书 | `docs/plans/` 或 `plan/` | 项目根目录 |
| 报告/说明 | `docs/` 或 `docs/reports/` | 项目根目录 |
| API文档 | `docs/api/` | 项目根目录 |
| 脚本工具 | `scripts/` | 项目根目录 |
| 配置文件 | `config/` | 项目根目录 |
| 测试文件 | `tests/` | 源代码目录（除非约定） |

### 新建文件的判断流程

```
新建文件
    ↓
┌──────────────────────┐
│ 文件是什么类型？      │
└──────────┬───────────┘
           ↓
    ┌──────┴──────┐
    │ 计划书？     │ → docs/plans/YYYY-MM-DD-名称.md
    │ 报告？       │ → docs/reports/YYYY-MM-DD-名称.md
    │ 代码？       │ → src/目录结构
    │ 测试？       │ → tests/
    │ 脚本？       │ → scripts/
    │ 配置？       │ → config/
    │ 文档？       │ → docs/
    │ 其他？       │ → 根据项目惯例或询问用户
    └─────────────┘
```

### 目录审阅检查

新建项目、改造工作流或新增文件前，必须先检查项目根目录是否具备以下基础结构：

| 必备项 | 用途 | 缺失时处理 |
|--------|------|------------|
| `docs/plans/` | 设计计划书 | 需要写计划时先创建 |
| `docs/reports/` | 分析、审查、验证报告 | 需要写报告时先创建 |
| `config/` | 配置文件 | 需要新增配置时先创建 |
| `scripts/` | 工具脚本 | 需要新增脚本时先创建 |
| `tests/` | 测试文件 | 需要新增测试时先创建 |
| `README.md` 或 `INDEX.md` | 项目入口说明 | 新项目必须至少保留一个入口说明 |

执行要求：
- 先审阅目录，再决定文件落点
- 只创建当前任务需要的目录和入口文件
- 不为了通过检查而创建空洞文档
- 不把计划、报告、脚本、配置、测试直接堆到根目录

### 特殊说明

- **已有项目**：跟随现有项目结构，不强制重构
- **新项目**：默认使用上述标准结构
- **单文件项目**：可以简化，但文档仍然放到 `docs/` 目录

## 三、技能智能路由

### 强制规则

**每次对话开始时，必须评估用户请求并智能选择合适的技能。不得跳过技能评估直接执行。**

### 路由流程

```
用户请求到达
    ↓
┌─────────────────────────────┐
│ 第一步：根本指令（root-directives）│  ← 你在这里
│ - 确保中文输出                │
│ - 确保项目结构规范             │
│ - 确保技能路由                 │
└──────────────┬──────────────┘
               ↓
┌─────────────────────────────┐
│ 第二步：任务评估（assessor）     │
│ - 评估复杂度：简单/中等/复杂    │
│ - 选择执行模式                 │
└──────────────┬──────────────┘
               ↓
┌─────────────────────────────┐
│ 第三步：技能匹配               │
│ - 根据任务类型匹配技能         │
│ - 加载匹配的技能               │
│ - 按优先级执行                 │
└─────────────────────────────┘
```

### 技能匹配规则

根据任务关键词自动匹配技能：

| 关键词/场景 | 技能 |
|-------------|------|
| 任何请求 | root-directives（根本指令） |
| 任何请求 | assessor（复杂度评估） |
| 需求不明确、多步骤 | task-planner（任务规划） |
| 头脑风暴、设计探索 | brainstorming（头脑风暴） |
| 写计划、实施方案 | writing-plans（编写计划） |
| 执行计划、开发任务 | plan-execution（执行计划） |
| 写代码、实现功能 | dev-expert（开发专家） |
| 代码审查、质量检查 | code-quality（代码质量） |
| Bug、调试、排错 | systematic-debugging（系统调试） |
| TDD、写测试 | test-driven-development（测试驱动） |
| 打造UI、设计界面 | ui-ux-pro-max 或 designer-expert |
| 完成前验证 | verification-before-completion |
| 提交代码 | git-commit-check（提交检查） |
| 删除文件 | safe-delete（安全删除） |
| 复杂架构设计 | architect-expert（架构专家） |
| 部署、CI/CD | devops-expert（运维专家） |

### 技能优先级

当多个技能可能适用时：

1. **root-directives** — 始终最先执行
2. **assessor** — 评估复杂度，决定模式
3. **流程技能**（brainstorming → writing-plans → plan-execution）— 按流程顺序
4. **质量技能**（code-quality, verification）— 执行后检查
5. **安全技能**（git-commit-check, safe-delete）— 操作前确认

### 复杂度评估触发

| 复杂度 | 模式 | 执行方式 |
|--------|------|----------|
| 简单 | quick-mode | 直接执行，无需规划 |
| 中等 | normal-mode | 简短规划（5分钟）+ 1-2专家 |
| 复杂 | deep-mode | 完整规划 + 多专家分阶段 |

## 四、上下文压缩规则

当进行上下文压缩时：
- 压缩总结必须使用中文
- 保留关键技术术语（命令、代码片段、文件路径）
- 不要翻译代码、命令、API名称

---

## 五、响应风格

- **简洁优先**：能用一句话说明的不用一段话
- **代码优先**：能直接展示代码的不先解释原理
- **中文注释**：代码中的注释用中文
- **结构化输出**：使用表格、列表、流程图组织信息

---

## 执行检查清单

每次对话开始时，快速检查：

```
□ 1. 是否使用中文输出？
□ 2. 是否先完成目录审阅检查？
□ 3. 需要新建的文件是否放到了正确的目录？
□ 4. 是否评估了应该使用哪个技能？
□ 5. 是否根据复杂度选择了正确的模式？
□ 6. 代码注释是否使用中文？
```

**此技能与 assessor 配合使用。assessor 负责评估复杂度，root-directives 负责基本规则。**
