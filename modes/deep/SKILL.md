# Deep Mode

## 目的

用于复杂项目、跨模块改造或新项目建设。先完整规划，再分阶段执行，并在关键节点进行质量检查。

## 适用场景

- 影响 4+ 个文件或创建新项目。
- 需要架构决策。
- 涉及多个组件或系统集成。
- 需求不清晰或持续变化。
- 预计 4 小时以上完成。

## 执行流程

```
用户请求
    ↓
阶段 1：需求与设计（PM、Architect、Designer）
    ↓
阶段 2：实现（Dev、QA 并行准备）
    ↓
阶段 3：质量保证（QA 深度审查）
    ↓
阶段 4：部署（DevOps）
    ↓
完成
```

## 响应模板

```
这是一个复杂项目，我会使用 Deep Mode 进行完整规划：

阶段 1：需求与设计
   PM：澄清需求
   Architect：设计系统
   Designer：设计 UI/UX

阶段 2：实现
   Dev：实现功能
   QA：准备测试

阶段 3：质量保证
   QA：代码审查与测试

阶段 4：部署
   DevOps：部署与运维检查

[开始阶段 1]
```

## 阶段细节

### 阶段 1：需求与设计

**时长：** 30-60 分钟  
**专家：** PM、Architect、Designer

输出：
- PM：功能规格、用户故事、验收标准。
- Architect：系统设计、API 合约、数据模型。
- Designer：线框图、设计系统、必要的高保真稿。

### 阶段 2：实现

**时长：** 根据范围决定  
**专家：** Dev、QA

并行工作：
- Dev：组件、API、业务逻辑实现。
- QA：测试用例准备。

同步点：
- API 合约确认。
- 集成测试确认。

### 阶段 3：质量保证

**时长：** 约占总时间 20%  
**专家：** QA

活动：
- 代码审查。
- 单元测试和集成测试。
- 安全检查。
- 性能检查。

### 阶段 4：部署

**时长：** 30-60 分钟  
**专家：** DevOps

活动：
- 环境配置。
- CI/CD 流水线。
- 监控配置。
- 回滚方案。

## 专家协调

### 通信流

```
PM ↔ Architect ↔ Designer
       ↕
      Dev
       ↕
      QA
       ↕
    DevOps
```

### 同步点

1. 阶段 1 后：全体设计审查。
2. 阶段 2 中：集成检查点。
3. 阶段 3 前：功能完成审查。
4. 阶段 4 前：上线或交付决策。

## 规则

1. 先完成规划，再开始实现。
2. 必要文档放入正确目录，不堆根目录。
3. 每个阶段完成后再进入下一阶段。
4. 不让专家孤立工作，关键节点同步。
5. 在方向性决策上与用户确认。

## 模式切换

### 降级到 Normal Mode

触发条件：
- 任务比初始判断简单。
- 早期出现明确解决方案。
- 不需要架构变化。
- 1-2 个专家即可完成。
- 预计少于 4 小时。

说明模板：

```
切换到 Normal Mode：原因
```

### 重新评估

触发条件：
- 范围明显变化。
- 出现新需求。
- 发现技术阻塞。
- 时间估算明显不准。

说明模板：

```
重新评估复杂度：原因
```

## 示例

### 适合 Deep Mode

- “构建电商平台。”
- “重构认证系统。”
- “设计实时聊天功能。”
- “创建新的移动应用。”
- “迁移到微服务。”

### 不适合 Deep Mode

- “修复这个 bug。” → 使用 Normal Mode。
- “怎么做……” → 使用 Quick Mode。
